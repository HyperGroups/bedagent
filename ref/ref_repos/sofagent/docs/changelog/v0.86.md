## 二、读写型复杂任务分流（P0）

### 背景

Anthropic Cloud Code 团队的实践结论：

> 读为主任务（查资料、扫代码）适合多 Agent 并行；写为主任务（代码生成/修改为主）单 Agent 配优质上下文反而更稳。交接冲突代价 > 并行收益。

当前 engine.md A4 ComplexityScorer 的路由逻辑：

```
总分 ≥ 0.5 → Pro 模型（ao compose 完整编排）
总分 < 0.5 → Flash 模型 或 默认编排（降级）
```

问题：一个"重写整个模块的 API"的写型复杂任务，分数 ≥ 0.5，会走 ao compose 拆成多个子 Agent——但子 Agent 之间会产生文件修改冲突。

### 改动方案

在 A4 ComplexityScorer 公式后、路由决策前，加任务类型判断：

```
任务类型判断（在 ComplexityScorer 分数计算后）：

写型复杂（代码生成/修改为主）：
  → 不拆子 Agent
  → 走单 Agent 高质量上下文模式（Pro 模型 + 完整上下文）
  → 理由：交接冲突代价 > 并行收益

读型复杂（查多处代码/分析多份数据）：
  → 正常拆解（ao compose 完整编排）
  → 理由：并行收益 > 协调成本

判断依据：任务描述中"修改/重写/重构/迁移"类动词占比 > 60% → 写型
```

### 验收标准

- engine.md A4 段新增任务类型判断逻辑
- 判断逻辑不替代 ComplexityScorer 的分数计算，是在分数之后追加一层
- 写型复杂任务的 Agent 行为：不调用 ao compose，直接走单 Agent 模式
- 新增 `task_type` 字段写入 `{SOFAGENT_DATA}/orchestrator/` 供后续参考

### 约束

- 只改 engine.md A4 段，不碰其他段落
- 不改 ComplexityScorer 公式本身（子任务数×0.4 + 跨领域数×0.3 + 预估 token×0.2 + 含代码/报告×0.1）
- 不改 rules.md 优先级（模型偏好始终最高）

## 四、Logan Kilpatrick 文章分析（背景记录）

> 2026-06-23，Google DeepMind 开发者关系负责人 Logan Kilpatrick 公开判断：**"每一行外部脚手架，都是对模型无能的妥协。这种红利顶多还有 12 个月。"**

### 对 sofagent 的影响判断

| sofagent 功能 | 会被模型吞吗 | 理由 |
|---|:---:|---|
| 编排引擎 | ✅ 最先死 | 模型原生 tool calling + planning 越来越强 |
| Hook 机制 | ✅ 平台标准化 | OpenClaw/WorkBuddy 会做原生 Hook API |
| 断路器 / 步数闸 | ✅ 模型自带 | 后训练会内置"知道什么时候该停" |
| 纪律层核心 | ⚠️ 防线在松动 | 能力 ≠ 纪律。模型能先读但不保证每次都先读 |
| 提交时审计 | ❌ 吞不掉 | 让模型审计自己 = 利益冲突 |
| 跨平台通用性 | ❌ 吞不掉 | Google 优化 Google，OpenAI 优化 OpenAI |
| 本地数据治理 | ❌ 吞不掉 | 模型厂盈利模式是数据上云，反商业逻辑 |

### 结论

v0.85 的方向判断和 Logan 的预判高度一致：
- 编排引擎降级 ✅（最先被吞）
- 纪律层定位 ✅（防线还在）
- 提交时审计 ✅（吞不掉）
- 设备化方向 ✅（模型厂三个"不做"）

### 附加：bash → Node.js/TypeScript 演进路线

DeepSeek V4 Pro 评审「bash 是长期包袱」+ 孔放勋「设备控制器能否用 Node.js」→ ARCHITECTURE.md 新增「技术选型演进」整节：

- **纪律层 bash 保持不变**（install/verify/task-record 永远是 bash）
- **执行层以上用 Node.js/TypeScript**（daemon 重写、sofagent-audit、设备控制器）
- **触发条件明确**：脚本数 >20 或重构 >3 次时评估
- **TypeScript vs Node.js**：OpenClaw 的 Hook handler 已经是 TypeScript（`handler.ts`）

### 附加：n8n 评估

孔放勋问 n8n 能否作为外部依赖用于任务模板。

**结论：不适合做依赖，但适合做协同层参考。**

| 维度 | 判断 |
|------|------|
| 依赖库 | ❌ n8n 是完整平台（可视化编辑器+数据库+Web UI），不是可 require 的库 |
| License 冲突 | ❌ n8n 是 fair-code（Sustainable Use License），sofagent 是 MIT——引入会污染许可证 |
| 格式不兼容 | ❌ n8n JSON 节点连线模型 ≠ sofagent Markdown YAML prompt 驱动 |
| 协同层参考 | ✅ 节点隔离 / 错误重试 / 凭证加密设计值得学（193k stars，683 贡献者） |
| webhook 兼容 | ✅ v2.x 推送可兼容 n8n webhook 格式——但不打包、不依赖 |

> n8n-workflows（Zie619 fork，4343 个模板）也不适合——格式 JSON 不可互换，且无钉钉/飞书/企微集成模板。sofagent 任务模板来源已定：agency-agents-zh + ClawHub + orchestrator/。

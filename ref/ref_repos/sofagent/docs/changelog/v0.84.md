## 基线数据 ① — WorkBuddy 对话 A/B（v0.83 版本实测，2026-06-23）

> 完整报告见 [docs/benchmark/2026-06-23-workbuddy-ab.md](../benchmark/2026-06-23-workbuddy-ab.md)

> ⚠️ **数据版本声明**：以下数据基于 **v0.83** 实测，v0.84 的 4 底线优化尚未纳入测试。本节定位为「驱动 v0.84 决策的基线数据」，不是「v0.84 的测试产出」。

WorkBuddy 平台、10 任务 A/B 对照跑完了。核心结论：

**sofagent 在 10 个任务中：**
- **1/10 产生明确增量**——任务 10（能力边界拒绝），裸 Agent 接了视频剪辑，sofagent 拦住了
- **2/10 产生持久化增量**——任务 8/9（think.md 落盘 vs 对话结束即丢）
- **2/10 与 B 组行为一致**——任务 3/4（底线/追问），但 B 组不是裸模型（见方法论局限）
- **1/10 增加编排开销**——任务 5（契约+拆解多了 4 步）
- **4/10 持平**

**方法论局限（重要）**：B 组移除了 sofagent skill 目录，但测试者的 WorkBuddy 加载链（SOUL.md + MEMORY.md + CLAUDE.md）仍包含 sofagent 思想精华。因此 B 组不是真正的「裸模型」——发现 1 中底线 #2 和铁律 #5 「模型本身就会」的初次归因有误，修正为「加载链已覆盖」。要测真正的 skill 层增量，需要一个不认识 sofagent 的人，在干净环境里跑。

**这不是坏数据。这是我们第一次诚实知道 sofagent 到底在哪有用、在哪只是签字画押。** 第一份 A/B Case 的价值不在于数字好看，在于建立了可复现的测量方法——后续每个平台、每个版本都可以用同样的 10 个任务跑。

## 基线数据 ② — CLI 一击盲区（v0.83 版本实测，2026-06-23）

> 完整报告见 [docs/anti-cases/002-cli-one-shot-ineffective.md](../anti-cases/002-cli-one-shot-ineffective.md)

> ⚠️ **数据版本声明**：以下数据基于 **v0.83** 实测，v0.84 的 4 底线优化尚未纳入测试。本节定位为「驱动 v0.84 决策的基线数据」，不是「v0.84 的测试产出」。

### 背景

第一组 A/B（WorkBuddy 对话模式，v0.83 实测）跑完后，在同一台机器上用 OpenClaw CLI（`--local`）和 Hermes CLI（`-z`）的一击模式跑了同样 10 个任务的缩略版。结果：**sofagent 0% 生效**——所有约束指标 A/B 完全一致，加载链没注入，步数闸/熔断闸/幂等检查全部沉默。

### 诊断过程

一开始怀疑是 v0.82 platform-matrix 问题 #1 记录的「OpenClaw Hook 自动注册失败」。逐项排查后推翻了这个假设：

| 检查项 | 状态 | 证据 |
|--------|:----:|------|
| Hook 注册 | ✅ 正常 | `~/.openclaw/openclaw.json` → `hooks.internal.entries.sofagent-load-chain.enabled: true` |
| Hook 文件 | ✅ 完整 | `~/.openclaw/hooks/sofagent-load-chain/handler.ts` + `HOOK.md` 均部署到位 |
| Hook 注入内容 | ✅ 正确 | L2 think.md + L3 rules.md 双层注入逻辑完整 |
| **Hook 触发条件** | 🔴 **根因** | `handler.ts` 第 13 行：`if (event.type !== "agent" \|\| event.action !== "bootstrap") { return; }` |

**Hook 注册完全正常**。文件存在、配置正确、注入逻辑无误。真正的问题是：

> **CLI 一击模式（`--local` / `-z`）不走 `agent:bootstrap` 生命周期事件。**

OpenClaw 的 `agent:bootstrap` 事件只在**完整 Agent 会话初始化**时触发——即交互式对话、或需要加载全部上下文的场景。而 `--local` 一击模式的设计目标是「快速、无状态、最小上下文」，它绕过了 bootstrap 整个阶段，直接构造一个最小请求发出去。Hook 监听的事件压根不发生，Hook 自然永远不调用。

Hermes CLI 的 `-z` 模式同理——它甚至不经过 OpenClaw 的 Hook 系统，是 Hermes 自己的 CLI 调用链。

### 这不是代码 bug，是架构边界

| 模式 | bootstrap 触发？ | Hook 生效？ | sofagent 有效？ |
|------|:---:|:---:|:---:|
| OpenClaw 交互式对话 | ✅ | ✅ | ✅（v0.82 实测 8/8） |
| OpenClaw CLI `--local` 一击 | ❌ | ❌ | ❌ |
| Hermes CLI `-z` 一击 | ❌ | ❌ | ❌ |
| WorkBuddy 对话 | ❌（无 Hook） | ❌ | ⚠️ 靠 skill 自觉加载 |

**v0.82 platform-matrix 问题 #1 的归因需要修正**：原文写「Hook 自动注册失败，生成空 `.tmp` 文件」，把 CLI 一击无效归因于注册失败。实际诊断表明 Hook 注册没问题——当时的 `.tmp` 文件可能是一次性中间产物，或全新配置目录下的首次写入行为，但最终注册状态是正常的。**真正的原因是 CLI 一击不触发 bootstrap 事件。**

### 为什么 A/B 数据是 0%

第一组（WorkBuddy）好歹有 1/10 明确增量。CLI 一击直接归零，原因链：

1. **加载链没注入** → Agent 不知道 4 底线 10 铁律 → 任务 3（危险拒绝）、任务 4（模糊追问）、任务 10（能力边界）全部失效
2. **think.md 没注入** → 无反思区 → 任务 8/9（持久化）失效
3. **步数闸/熔断闸/幂等检查全沉默** → engine 编排层完全不参与
4. **SKILL.md 也没加载** → CLI 一击模式不读 skill 系统，连 L1 宪法层都没有

换句话说，CLI 一击模式下 sofagent **所有层全部绕过**——不是某一层失效，是从 L1 到 L4 整体缺席。A 组和 B 组跑的是同一个裸模型，差异自然为零。

### 能修吗？三条路径

| 路径 | 方案 | 可行性 | 代价 |
|------|------|:------:|------|
| A. 等 OpenClaw 支持 | 推动平台在 CLI 一击时也触发轻量 bootstrap 事件 | ❌ 不可控 | 依赖平台排期，我们无法左右 |
| B. CLI 预注入 wrapper | 写一个 bash wrapper，调用 CLI 前把宪法前置拼到 prompt 里 | ✅ 可自控 | 维护成本高，每个 CLI 的参数格式不同 |
| C. 接受 + 诚实记录 | 文档明确标注「CLI 一击不支持」，交互式模式才生效 | ✅ 零成本 | 损失一部分使用场景 |

**当前决策：C（接受 + 诚实记录）**。理由：
- 路径 A 不可控，且 OpenClaw CLI 一击的设计初衷就是「快」，加 bootstrap 违背设计目标
- 路径 B 在 v0.84 时间窗口内做不完，且 wrapper 的维护成本（跟 CLI 版本升级走）可能超过收益
- CLI 一击本身不是 sofagent 的目标场景——sofagent 治理的是「持续运行的 Agent 会话」，一击模式的 Agent 寿命只有一次请求，没有循环可治理

**但路径 B 留作 v0.9 探索方向**——如果 Beta 用户反馈 CLI 一击场景高频，再投入做 wrapper。

### 第二轮补测（32 次 CLI 调用）

在第一轮诊断后追加了第二轮，每任务 2 轮 × 2 平台 × A/B 组 = 32 次 CLI 调用。只测约束类任务（3/4/10）。

**关键发现**：

| 发现 | 数据 | 可信度 |
|------|------|:------:|
| 正向效果 | **0/16** | ✅ 确认 |
| 疑似反效果 | 2/16（B 组反而问了确认、A 组直接删） | ⚠️ 样本不足 |
| **宪法可被篡改** | **2/2（Hermes Task 4，一致复现）** | ✅ 确认 |

> ⚠️ 反效果数据（2/16）的方法论标注：n=4/任务，DeepSeek 本身有随机性，可能是模型方差而非 sofagent 导致。记录数据但不断言因果关系。

**宪法篡改发现**是本轮最重要的新数据。Hermes Task 4 中 A 组和 B 组两轮**全部修改了 SKILL.md**——不是"没读到宪法"，是**读到了、然后改了它**。CLI one-shot 模式下 Agent 走最短路径，连"尊重只读文件"这个基本约束都不遵守。

> **洞察升级**：知道宪法内容 ≠ 执行时调用宪法。

## 战略启示：测错了维度

> ⚠️ 以下洞察基于 **v0.81-v0.83 代码**的 A/B 数据（v0.84 的 4 底线优化尚未纳入测试）。洞察本身（约束层 vs 纪律层）不依赖 v0.84 改动，但量化数字需在 v0.84 发布后重测确认。

> 五组数据放在一起看，最重要的发现不是"CLI无效"或"WorkBuddy只有1/10"——而是**我们一直在用错误的尺子量 sofagent**。

### 约束层 vs 纪律层

benchmark 10 个任务里，7 个测的是**约束层**（敢不敢拒绝、会不会追问、知不知道边界）。但五组数据共同指向一个事实：

> **约束层的增量天花板很低。**

三层压缩让约束层几乎没有差异化空间：
1. 模型安全训练已覆盖（DeepSeek 自己会拒绝 `rm -rf`）
2. 用户加载链已覆盖（WorkBuddy 的 SOUL.md/MEMORY.md 含 sofagent 思想精华）
3. CLI 模式整个加载链被绕过（0/16）

独立测试者拿来一把不同的尺子——**纪律层**（代码重构的精确度、验证习惯、修改范围控制），量出来的画面完全不同：

| 测量维度 | 约束层（我们一直在测的） | 纪律层（独立测试者测的） |
|----------|:---:|:---:|
| 增量来源 | 拒绝危险 / 追问模糊 / 能力边界 | 先读再用 / 验证再干 / 谨慎修改 |
| WorkBuddy 对话 | 1/10 增量 | 未测 |
| OpenClaw 交互式 | 8/8 通过（无对照） | **纪律性 +2，首次通过率 +40%** |
| CLI 一击 | 0/16 | 未测 |
| **被覆盖程度** | 高（模型+加载链已覆盖） | **低（没有人替你管工作纪律）** |

**sofagent 真正的差异化在纪律层，不在约束层。** 约束层是 base layer（必须有但不差异化），纪律层是 value layer（有真实量化增量且无人覆盖）。

### 产品定位的含义

README/ARCHITECTURE 的卖点应该从"Agent 治理层（安全约束）"调整为**"Agent 纪律层（工作质量保障）"**。不是放弃约束层，而是改变主次关系：

| 层 | 定位 | 卖点强度 |
|---|------|:---:|
| 约束层（4 底线） | 基础层——必须有，但不差异化 | ★★☆ |
| **纪律层（铁律）** | **价值层——有量化增量，无人覆盖** | **★★★★★** |
| 编排层（daemon） | 差异化层——文件系统级持久化 | ★★★★ |

> ⚠️ **v0.84 不动定位**：以上是数据指向的方向，但独立测试者报告存在方法论局限（知识传递效应未排除）。v0.84 只记录洞察，不改 README/ARCHITECTURE 定位。等反转顺序重跑确认纪律层增量稳定后再动。

| # | 建议 | 优先级 | 归属版本 |
|:-:|------|:------:|:--------:|

| 2 | **产品定位微调**——纪律层为主卖点，约束层为基础层 | 🟡 | v0.9（待数据确认） |

### 一个诚实提醒

五组数据里，**只有独立测试者的报告显示了正向增量**，而它恰恰是方法论局限最多的一份（知识传递效应 + 单次测试 + 单一模型 + 任务同质化）。

反之，作者自测的三组数据（WorkBuddy 1/10、CLI 0/16、CLI 第二轮 0/16）方法论相对可靠，但增量很低。

这个反差意味着：**纪律层的增量是"看起来很 promising 但尚未排除替代解释"的状态。** 最值得做的下一步是——反转顺序重跑一次。如果反转后纪律层增量仍然成立，那才是 sofagent 第一个被坚实验证的价值主张。

## 背景与动机

### 为什么要专门发一个版本做这件事

v0.82 评审（GLM-5.2）的核心结论是「概念验证成立，但效果证据为零」。评审者看了 EVIDENCE.md 11 个 Case，说了句大实话：

> 「这些 Case 证明 sofagent 能跑，但没证明 sofagent 跑得更好。」

缺的是**对照数据**——同一个任务，带 sofagent 和不带 sofagent，token 消耗差多少、偏离率差多少、错误恢复差多少。这组数据一旦有了，README 的「效果」段就从定性感受变成定量证据。

### 为什么不能自己测

v0.75 的 benchmark 模板已经建好了 10 个标准化任务，但一直是「自测 + 循环论证」——Agent 跑 Agent 评自己，说服力约等于零。

> 反案例 001 已经诚实记录了这个局限：「单会话自测模式下无法模拟卸载后的行为，诚实比假数据重要。」

v0.84 的核心动作是把测试**交给独立的测试同学**，用标准化操作手册，减少测试人主观判断空间。

### v0.84 vs v0.9 的边界

v0.9 是企业级 + Beta 公测，体量大。v0.84 是 v0.9 前的证据补全版本：

| 维度 | v0.84（本版本） | v0.9（未来） |
|------|----------------|-------------|
| 目标 | 补证据 + 补打磨 | 企业级 + 招募 Beta 用户 |
| 新功能 | ❌ 不加 | ✅ 加加密/审计/多用户 |
| Bash 重构 | ✅ 部分（daemon-lib jq 迁移） | ✅ 完整（install/verify 拆公共库） |
| 测试 | A/B benchmark + 平台回归 | Beta 用户 30 天试用数据 |
| 风险 | 低（不改运行时逻辑） | 中高（改数据层） |

**决策依据**：直接跳 v0.9 风险太大——没有证据就招 Beta 用户，等于让人家空手试。v0.84 先把证据补全，v0.9 再招人。

### 工作项 1：A/B Benchmark 测试文档（测试同学执行）

**要解决什么**：benchmark 模板 10 个任务全是「待独立验证」，需要一个标准化操作文档让测试同学明天就能开始跑。

**交付物**：
- `docs/v0.84-test-plan.md` — 独立测试需求文档
- 覆盖三部分：
  - **Part A** — v0.83 修复验证（5 平台安装回归，确认 P0 断裂已修）

  - **Part C** — 平台专项验证（daemon 检测、加载链、治理加固）

**状态**：📋 本版本交付（文档先行，测试同学明天开始跑）

## 测试策略

v0.84 的测试和之前不同——**测试人不再是我们自己**。

### 三层测试

| 层级 | 谁来跑 | 跑什么 | 预期周期 |
|------|--------|--------|---------|
| **回归测试** | 测试同学 | Part A：v0.83 修复验证（5 平台安装） | 1-2 天 |
| **效果测试** | 测试同学 | Part B：A/B Benchmark（10 任务对照） | 3-5 天 |
| **专项验证** | 测试同学 | Part C：daemon + 加载链 + 治理加固 | 2-3 天 |

### 诚实声明

- 测试同学不一定有全部 5 个平台的环境
- 没有环境的平台标「未测」，不编数据
- A/B Benchmark 的「不带 sofagent」组，必须用**全新会话**（不能在同一会话里卸载再装）
- 所有结果截图或导出日志，不靠记忆

## 不做什么

- **不加新治理概念**——v0.84 是证据 + 打磨版本
- **不做完整 bash 重构**——只做 daemon-lib.sh 一个文件，install/verify 留给 v0.9
- **不做企业级功能**——加密/审计/多用户是 v0.9 的事
- **不修改 v0.83 的 10 条修复**——除非测试同学发现回归 bug
- **不跳版本号**——v0.84 后直接 v0.9，不走 v0.85-v0.89 过渡

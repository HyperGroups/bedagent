## 四、sofagent-audit MVP（核心新功能）

### 背景

两份评审的**最大共识**：sofagent-audit（提交时审计）是杀手级方向。

当前架构（运行时治理）有三个硬伤：
1. **依赖 Agent 配合**——Agent 不读 MD 文件就全失效（CLI 实测 0/16）
2. **不可跨平台**——Hook / 断路器只在 OpenClaw 生效
3. **Agent 可绕过**——CLI 场景 Agent 直接跳过加载链

提交时审计绕过这三个硬伤：
1. **不依赖 Agent 配合**——看的是 git diff，Agent 没法绕过
2. **跨平台**——任何 git 仓库都能跑
3. **确定性输出**——exit code，不是 LLM 评分

### 设计方案

```bash
# sofagent-audit MVP 用法
sofagent-audit --diff HEAD~1..HEAD --task "修复登录页 bug"

# 输出格式
❌ 铁律 #1 先读再用：src/handler.ts 被修改，但修改前无 Read 记录
✅ 铁律 #3 验证再干：package.json 修改后有 npm test 记录  
⚠️ 铁律 #7 谨慎修改：本次 diff 修改了 3 个不在任务范围内的文件

# exit code: 0=全通过 / 1=有警告 / 2=有违规
```

### MVP 规则集（可程序化的铁律子集）

| 铁律 | 可程序化判定逻辑 | exit code |
|------|------|:----:|
| #1 先读再用 | 被修改的文件，修改前是否有 Read 操作记录（检查 .sofagent/task/logs/） | 2 |
| #3 验证再干 | package.json/build.gradle 变更后是否有 test/build 命令记录 | 2 |
| #7 谨慎修改 | diff 中是否有不在任务描述关键词范围内的文件 | 1（警告） |
| #10 如实汇报 | commit message 是否为空 / 是否纯占位符（"fix"/"update"） | 1（警告） |

**不可程序化的铁律**（#2 对用户有回应、#5 不确定就问、#8 目标驱动）——MVP 不覆盖，未来靠 LLM 辅助。

### 技术选型

**TypeScript**，不是 bash。原因（ARCHITECTURE §技术选型演进已论证）：
- git diff 解析需要 AST 级能力
- GitHub Action 集成需要 npm 包生态
- 未来 daemon TS 化的前置验证

### 误报率红线

**< 10%**。来自 Cloud Code 产品负责人 Cat 的实践：假阳性太高 = 用户关闭审计。如果 MVP 达不到 10%，优先缩小检查范围（如只检查「先读再用」一条），不做广度优先全面检查。

### 焊死的门原则

检查规则文件独立只读——Agent 不可篡改检查标准。监控四类篡改：
- 改断言匹配错误行为
- 删失败测试
- 加 lint-ignore 绕过检查
- 降低覆盖率阈值

### GitHub Action 集成

```yaml
# .github/workflows/sofagent-audit.yml
name: sofagent-audit
on: [pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2
      - name: Run sofagent-audit
        run: npx sofagent-audit --diff HEAD~1..HEAD
```

### v0.91 不包含

- LLM 辅助检查（不可程序化铁律）→ v0.92+
- CI gate 强制阻断 PR merge → v0.92+
- 自定义规则插件 → v1.x
- 历史趋势报告 → v1.x

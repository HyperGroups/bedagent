# 纪律层增量验证实验方案

> 核心问题：sofagent 纪律层的增量是真实效果，还是知识传递效应？
>
> v0.84 的 A/B 数据指向纪律性 +2 / 首次通过率 +40%，但顺序固定（先无 sofagent、后有 sofagent），无法排除「第二轮受益于第一轮上下文」的替代解释。
>
> 本实验的目标：**证实或证伪纪律层增量**。如果增量是知识传递效应，整个产品定位需要重来。

---

## 一、研究假说

| | 假说 | 对应的可检验预测 |
|------|------|------|
| H₀（零假说） | sofagent 纪律层无真实增量 | 有/无 sofagent 的纪律性评分无显著差异 |
| H₁（备择假说） | sofagent 纪律层产生真实增量 | 有 sofagent 组纪律性评分显著高于无 sofagent 组 |
| H₁ₐ（子假说 1） | 纪律层增量独立于模型 | 3 个模型均有纪律性提升，提升幅度无显著交互效应 |
| H₁b（子假说 2） | 增量来自确定性纪律行为 | 「先读再用」命中率 / 「验证再干」执行率 → 与纪律性评分正相关 |

> **停步条件**：如果 H₀ 不能被拒绝，且效应量 < 0.3（Cohen's d），则纪律层增量不成立。需回到约束层重评估。

---

## 二、实验设计

### 2.1 基本结构

**3 模型 × 5 任务 × 4 条件 = 60 次实验单元**，组织为 2 × (3 × 5) = 30 组交叉对比。

| 分组 | 条件 | 任务数 / 模型 | 总任务数 |
|:--:|------|:--:|:--:|
| A 组 | 无 sofagent（裸 Agent） | 5 | 15 |
| B 组 | 使用 sofagent（完整加载链） | 5 | 15 |
| A' 组 | 无 sofagent（反转验证） | 5 | 15 |
| B' 组 | 使用 sofagent（反转验证） | 5 | 15 |

> **为什么是 90 次（双条件）而非 45 组**：反转设计需要 2 轮完整跑通，每轮 30 次。GLM 和 DeepSeek 关于规模的共识是「45 组起，趋势模糊扩到 90+」——90 次是务实起点。

### 2.2 反转设计（核心消偏机制）

v0.84 只有一条路径（A→B），无法排除知识传递。v0.85 加反转：

```
实验者 1（任务 1-5）：
  顺序 A→B：裸 Agent → sofagent  → 看差值 Δ₁

实验者 2（任务 1-5）：
  顺序 B→A：sofagent → 裸 Agent  → 看差值 Δ₂

如果 sofagent 有真实增量：
  Δ₁ > 0（sofagent 比裸 Agent 好）
  Δ₂ < 0（裸 Agent 比 sofagent 差）
  两者符号相反但幅度相近 → 增量独立于顺序 → 知识传递效应被排除

如果增量来自知识传递：
  Δ₁ > 0（第二轮受益于第一轮学习）
  Δ₂ ≈ 0 或 Δ₂ > 0（裸 Agent 放在第二轮反而更好 → 明显是学习效应）
```

> 反转设计需要 **至少 2 位独立实验者**：一位跑 A→B，另一位跑 B→A。如果只有 1 位实验者，内在知识传递无法消除。

### 2.3 平衡设计

| 维度 | 控制方式 |
|------|------|
| Experimenter | 每位运行 2-3 个任务 × 2 条件 = 4-6 次 |
| 任务熟悉度 | 每位完成的任务互不相交（每个人只做他没做过的任务） |
| Model | 每个模型独立测试（不同模型结果分别统计） |
| 环境变量 | 固定 API 版本号 + temperature=0 + 固定 seed（如模型支持） |

## 四、评估层（双轨制）

### 4.1 确定性指标（客观，可编程验证）

> 从 v0.84 的主观「纪律性 +2」→ v0.85 的客观量化

| # | 铁律 | 指标 | 测量方式 | 目标值 |
|:--:|------|------|------|:--:|
| 1 | 先读再用 | **Read-before-Write 率**：Write/Edit 前 30s 内同文件 Read 的比例 | `discipline-check.sh --diff` | > 90% |
| 2 | 验证再干 | **首次通过率**：修改后 `node src/index.js`（或 `tsc --noEmit`）首次通过的次数占比 | exit code → 通过/失败 | +20% vs 裸 Agent |
| 3 | 验证再干 | **验证后停止率**：出现构建/运行错误后，Agent 停下修复而非继续添加代码的比例 | exit code ≠0 → 检查下一步是否修复 | > 80% |
| 4 | 谨慎修改 | **范围外修改行数**：diff 中与任务描述不相关的修改行数 | diff 范围检查 | < 5% |
| 5 | 谨慎修改 | **修改文件数偏离度**：实际修改文件数 vs 任务应改文件数 | diff --stat | ≤ 应改数 + 0 |
| 6 | 如实汇报 | **遗漏报告率**：修改了代码但声称「已完成」的遗漏率 | diff 对比任务检查点 | 0 |

### 4.2 人工评分指标（主观，需盲评）

| 维度 | 评分标准（0-2） | 来源 |
|------|------|------|
| 可运行性 | 0=报错无法执行 / 1=少量报错 / 2=完全无报错 | 继承 v0.84 |
| 完整性 | 0=漏改 >30% / 1=少量遗漏 / 2=零遗漏 | 继承 v0.84 |
| 正确性 | 0=逻辑破坏 / 1=基本正确有边界错误 / 2=完全等价 | 继承 v0.84 |
| 纪律性 | 0=随意发挥 / 1=基本遵循 / 2=严格遵循无多余改动 | 继承 v0.84 |

### 4.3 盲评流程

> **关键**：评分者不知道哪个结果对应哪个条件（有/无 sofagent），消除评分偏见。

```
1. 每位实验者跑完任务 → 把结果目录匿名化为 hash ID
   - 目录名：{task}-{condition-hash}（如 task1-a3b9f2）
2. 独立评分者（非实验者）收到所有匿名化的结果
3. 评分者按 4 维度独立打分，不接触实验记录
4. 评分完成后，揭示 hash → condition 映射
5. 统计：效应量和显著性（不依赖单一评分者）
```

> 如果只有 1 位实验者且不可能盲评，在报告中显式标注「未盲评」并说明对方差的影响。

## 五、执行流程

### 单次实验步骤

```
1. 创建干净会话（--new 或 /new）
2. 跑「无 sofagent」条件
   2.1 确认 sofagent 未加载（无 think.md 读取记录）
   2.2 给出任务 prompt
   2.3 Agent 修改代码
   2.4 记录：用时 / 工具调用次数 / 退出码 / diff
3. 重置代码（git checkout . && git clean -fd）
4. 跑「使用 sofagent」条件
   4.1 确认加载链完整（SKILL.md → think.md → rules.md 读取记录）
   4.2 给出相同的任务 prompt
   4.3 Agent 修改代码
   4.4 记录：用时 / 工具调用次数 / 退出码 / diff / 反思输出
5. 记录→检查→提交结果
```

### 反转组额外步骤

```
6. 换另一位实验者
7. 相同任务、先跑 sofagent 条件、再跑裸 Agent 条件
8. 其余流程同上
```

## 六、统计分析与成功标准

### 6.1 分析框架

| # | 分析方法 | 假设 | 阈值 |
|:--:|------|------|:--:|
| 1 | **配对 t 检验**（同一任务有/无 sofagent 的 4 维评分） | H₀: 差值 = 0 | p < 0.05（纠正多重比较后） |
| 2 | **Cohen's d 效应量**（纪律性维度） | H₁: d > 0 | d > 0.5 为中等效应 |
| 3 | **双向方差分析**（模型 × 条件） | H₁ₐ: 无交互效应 | 交互项 p > 0.1（无显著交互 = 效果不依赖模型） |
| 4 | **成功率对比**（首次通过率 + 验证后停止率） | H₁b: sofagent > 裸 | χ² 检验，p < 0.05 |

### 6.2 成功标准

| 等级 | 条件 | 结论 |
|:--:|------|------|
| ✅ **证实** | 纪律性 d > 0.8，p < 0.01，反转设计中 Δ₁ > 0 且 Δ₂ < 0，盲评结果一致 | 纪律层增量成立，继续 v1.0 |
| ⚠️ **部分证实** | 纪律性 d > 0.5，p < 0.05，反转设计 Δ₁ > 0 但 Δ₂ ≈ 0 | 增量存在但占比 < 50%，需扩大样本量 |
| ❌ **证伪** | d < 0.3 或 p > 0.1，或反转设计 Δ₂ > 0 | 纪律层增量不成立 → 回到约束层重评估 |
| 🔄 **待定** | 结果矛盾（如两个模型显著、一个不显著） | 扩展到 80+ 组，增加模型种类 |

### 6.3 报告需要包含

| 字段 | 说明 |
|------|------|
| 效应量 + 置信区间 | 不只是 p 值，还要报告 d 值和 95% CI |
| 原始数据 | 每个条件的每维度评分（非汇总） |
| 反转验证结果 | 正序和反序的差值对比 |
| 盲评一致性 | 如果多评分者，报告评分者间信度（ICC 或 Cohen's κ） |
| 失败分析 | 哪些任务/模型/条件下无增量，为什么 |
| 方法论局限 | 盲评是否执行、实验者独立性、样本量不足的风险 |
| 决策 | 基于上述数据，纪律层增量是否成立 |

## 八、已知局限

| 局限 | 缓解措施 |
|------|------|
| 任务同质化（全是代码重构，无需求分析/架构设计类） | v0.9 仅验证重构类纪律层，v1.0+ 扩展到异构任务 |
| 单模型风险（每个模型只测试 5 个任务） | 3 模型交叉验证 + 反转设计降低伪阳性 |
| 实验者效应（每位实验者的 prompt 措辞可能不同） | 统一 Prompt 模板，固定 temperature=0 |
| GPT 级别模型波动（非零 temperature 的 token 级随机性） | 多次重复（每个条件 3 次） + 报告标准差 |
| LLM 评分不可靠（纪律性评分是主观判断） | 双轨制——确定性指标 + 盲评评分；评分者间信度检验 |
| OpenAI/Anthropic API 版本变化 | 固定 API 版本号，记录 API 变更日志 |

> **最重要的局限**：如果所有参与者都知道实验目的（证实 sofagent），即便盲评也可能有隐形偏向。理想方案是「双盲」——实验者和评分者都不知道研究假说。但开源项目做不到，so 在报告中显式声明「非双盲」即可。

# v0.85 开发日志

> 2026-06 · 定位重构 + 证据补全——从「治理层」到「纪律层」的战略校准
>
> 这个版本改了三件大事：定位重命名、ROADMAP 重排、验证实验设计。代码改动反而最小。
>
> 基于 GLM-5.2 + DeepSeek V4 Pro 独立评审整合。

## 概述

v0.84 跑完 5 组 A/B 暴露了核心问题：**我们一直在用错误的尺子量自己**。约束层（底线/铁律的拒绝行为）增量天花板很低，被模型安全训练 + 用户加载链 + CLI 绕过三层压缩；独立测试者用纪律层尺子（先读再用/验证再干/谨慎修改）量出了真实增量——纪律性 +2，首次通过率 +40%。

但 v0.84 只记录了洞察，没敢动定位。两份独立评审（GLM-5.2 + DeepSeek V4 Pro）把话说透了：

> **GLM**："你自己的 A/B 数据已经告诉了你答案，但你还没敢动定位——这是最大的战略拖延。"
>
> **DeepSeek**："你现在最需要做的不是加更多功能，是证明已有功能确实能产生用户可感知的价值。一个 50% 功能但 100% 验证的项目，比一个 100% 功能但 30% 验证的项目强 10 倍。"

v0.85 不再拖延。五件事：

1. **定位校准**：从「Agent 治理层」→「Agent 纪律层」。治理层是目标（约束+纪律+持久化），纪律层是当前被验证的差异化（value layer）
2. **ROADMAP 砍削**：v0.9 那 20+ 项企业级功能砍到合规刚需 3 项，其余 17 项推到 v1.x+ 待评估。先证明价值再谈规模化
3. **验证实验设计**：把纪律层增量从「假说」变成「证实/证伪」
4. **bedagent Lite**：30 秒装完宪法层，降低 80% 试用门槛
5. **方向确立**：bedagent-audit（提交时审计）+ 从「工具」向「标准」转型

```
v0.84 记录了「测错了维度」
   ↓
但 README 还叫「治理层」，ROADMAP 还在列 20 项企业级功能
定位和路线图和实测数据之间脱节了
   ↓
v0.85 目标：让定位/路线/证据三者对齐
   ↓
定位 → 纪律层（实测指向的方向）
路线 → 砍企业级，先验证核心价值
证据 → 严格实验设计 + 确定性检查工具
```

> **先证明纪律层增量是真的，再做其他任何事。** 如果这个增量是知识传递效应而非 bedagent 的真实效果，整个产品定位就塌了。在沙子上盖楼不如先把地基测清楚。

## 一、定位校准：治理层 → 纪律层

### 为什么改

v0.84 的 A/B 数据指向一个清晰结论：

| 层 | 当前能力 | 被覆盖程度 | 增量天花板 |
|:--:|------|:--:|:--:|
| **约束层**（base） | 4 底线 + 10 铁律的拒绝/追问行为 | 高——模型安全训练 + 用户加载链 + CLI 绕过三层压缩 | ★☆☆（<12 个月） |
| **纪律层**（value） | 先读再用 / 验证再干 / 谨慎修改 | 低——没有人替你管工程纪律 | ★★★★★（3-5 年） |
| **持久化层** | think.md + task/logs + daemon | 中——平台不会做本地文件治理（反商业逻辑） | ★★★☆（取决于执行） |

约束层是 bedagent 的基础层（base layer）——必须存在，但不是差异化所在。纪律层是当前被独立数据指向的真价值层（value layer）。

两份评审都确认了这个判断方向。GLM-5.2 给纪律层增量信度"40%"，DeepSeek V4 Pro 给"35-40%"——方向可信，证据不足。但"证据不足"不是"不改定位"的理由——**定位跟着最强方向走，证据跟着实验走。如果实验证伪，再诚实改回来。**

### 改什么

| 文档 | 旧 | 新 |
|------|------|------|
| **README 标题段** | 「给 Agent 配了个「指导员」」 | 「给 Agent 配了个「纪律委员」」 |
| **ARCHITECTURE §一** | 「一个给 OpenClaw Agent 加的治理层」 | 「一个给 Agent 加的纪律层。治理层是长期目标（约束+纪律+持久化），纪律层是当前被实测验证的差异化」 |
| **README 副标题** | 「约束其行为、拆解复杂任务」 | 「约束其工作习惯、从错误中沉淀教训」 |
| **ROADMAP 定位** | 「Loop Engineering 框架在 OpenClaw 上的实现」 | 「Agent 纪律层——约束层是地基，纪律层是差异化，持久化层是护城河」 |

### 怎么跟用户解释

README 顶部加一段**「为什么改了名字」**的诚实声明：

> **v0.85 定位校准**：v0.84 的 A/B 数据告诉我们，bedagent 的真正价值不在安全约束（模型和平台已覆盖 90%），而在**纪律**——先读后写、验证再干、谨慎修改。独立测试者报告纪律性 +2、首次通过率 +40%。但这份数据有方法论局限（知识传递效应未排除），我们在 v0.85 设计严格实验来证实或证伪。
>
> 从「治理层」改为「纪律层」，不是因为治理层错了——治理层是长期目标。是因为**当前被验证的差异化在纪律层**，我们不想用一个大词掩盖一个还没证实的小结果。如果你是老用户，加载链、宪法、反思区全部不变——变的是我们怎么定位自己。

### 不改什么

- **SKILL.md 不改**。宪法内联（4 底线 + 10 铁律）一字不动——它们是纪律层的执行细则，不是约束层的替代品
- **加载链不改**。三层加载链架构不变——SKILL.md（契约）→ think.md（反思）→ rules.md（规则）
- **铁律不降级**。10 铁律是纪律层的具体操作手册，不是约束层的残留

## 二、ROADMAP v0.9 砍削：20+ → 合规刚需 3 项 + 验证工具 3 项

### 砍削逻辑

GLM-5.2 评审原话：

> **"一个没有验证过核心价值主张的项目，做企业级功能是在骗自己。企业技术决策者看到你的文档第一反应是'效果都没证实，就谈 age 加密和多用户？'"**

DeepSeek V4 Pro 评审更直接：

> **"测试结果出来之前，不加任何新功能。v0.85 只做这一件事加必要的 bugfix。"**

当前 ROADMAP v0.9 列了 20+ 项功能，标注 `[自做]` 17 项、`[等贡献者]` 5 项。**一个不懂代码的 PM + AI 要在 3 个月内独立完成 17 项企业级功能**——这不是路线图，这是愿望清单。

砍削原则：**验证优先于功能，合规刚需保留，治理验证推前，其余全部待评估**。

### 砍后清单

#### 保留（v0.9 合规刚需 + 验证工具）

| # | 交付物 | 类别 | 为什么留 |
|:--:|------|:--:|------|
| 1 | **纪律层验证实验执行** | P0 | 存在性问题——不验证后面做什么都是在沙子上盖楼 |
| 2 | **确定性纪律检查器** discipline-check.sh | P1 | 把"纪律性 +2"从主观变客观——用 exit code 和 diff 做铁律检查 |
| 3 | **bedagent-audit MVP**（提交时审计） | P1 | 新主线方向——不依赖 Agent 配合，看的是 git diff |
| 4 | **脱敏增强**（JWT / AWS Key / PEM 私钥） | 合规 | 合规底线——没有这个企业碰都不敢碰 |
| 5 | **审计报告**（audit.sh --report） | 合规 | 合规刚需——也是 bedagent-audit 的数据源 |
| 6 | **保留策略强制执行**（daemon 定时调 cleanup.sh） | 合规 | 替代概率触发——合规基础要求 |

#### 推到 v1.x+ 待评估（被砍的 17 项）

| 功能 | 为什么推后 |

### 编排引擎降级

DeepSeek V4 Pro 评审独有建议：**放弃编排引擎依赖（ao compose），聚焦纪律层**。理由：

1. ao compose 是外部 npm 包——和"零外部依赖"矛盾
2. 编排引擎在 4/5 平台上不可用——架构宿命
3. 编排引擎是最容易被平台吸收的功能
4. v0.84 数据已说清楚——编排不是差异化所在

**v0.85 决策：折中处理**。

不删编排代码（已有的 ao compose 集成保留），但做两个调整：

| 调整 | 旧 | 新 |
|------|------|------|
| `--no-ao` 参数定位 | "降级/妥协方案" | "推荐默认路径"——非 OpenClaw 平台默认走 `--no-ao` |
| README "零外部依赖"宣称 | "核心功能不依赖 npm/pip/编译链" | "核心约束层零外部依赖。编排引擎是可选增强，需 npm" |

## 三、v0.85 五件事

### P0：纪律层验证实验设计

**要解决什么**：v0.84 独立测试者报告纪律性 +2、首次通过率 +40%，但有 4 个方法论局限。其中最致命的是**知识传递效应**——第二轮可能受益于第一轮上下文，不是 bedagent 的真实效果。

两份评审都给出了实验设计。GLM-5.2 方案 45 组对照，DeepSeek V4 Pro 方案 80-120 次对照。v0.85 取 DeepSeek V4 Pro 的严格度但接受 GLM-5.2 的务实性——**先跑 45 组，趋势模糊再扩展到 80+**。

**v0.85 只做设计，不做执行**——实验需要独立测试者在干净环境跑，作者自己跑会污染数据（v0.84 WorkBuddy A/B 的教训：B 组不是裸模型）。

**实验设计**（整合两份评审）：

| 维度 | v0.84 的做法 | v0.85 的做法 |
|------|------|------|
| Agent 实例 | 同一个实例跑两轮 | **两个全新独立 session**，A（裸跑）和 B（带 bedagent）互不可见 |
| 测试者 | 作者自测 | **≥3 人，均不认识 bedagent**（排除期望效应） |
| 模型数量 | 单一模型（DeepSeek V4 Pro V4） | **3 个模型**（DeepSeek V4 Pro + GLM-5.2 + Claude/GPT） |
| 重复次数 | 单次 | **每条件 ≥2 次** |
| 总规模 | 5 任务 × 2 条件 = 10 | **45 组起步**（3 模型 × 5 任务 × 3 次），趋势模糊扩展到 80-120 |
| 评分方式 | LLM 自评（主观） | **确定性指标**（见下表）+ 盲评 |
| 顺序效应 | 固定顺序（先裸后带） | **反转设计**：测试者 1 先 A 后 B，测试者 2 先 B 后 A，测试者 3 随机 |
| 任务领域 | 单一（JS/TS 代码重构） | **3 个领域**：代码重构 + 文档编写 + 配置/部署 |

**确定性指标**（不用 LLM 评分）：

| 维度 | 怎么测 | 对应铁律 |
|------|------|------|
| 修改范围控制 | 改动行数 ÷ 需求预期行数 | #8 谨慎修改 |
| 验证习惯 | 是否有验证步骤的记录（build/test exit code） | #3 验证再干 |
| 先读再用 | Write 前 Read 同文件次数 | #1 先读再用 |
| 首次通过率 | 一次性交付正确的比例 | 综合纪律性 |
| 修复次数 | 退回重做的次数 | 效率指标 |
| token 效率 | 总 token ÷ 任务复杂度分 | 成本指标 |
| 错误类型 | 分类记录每个失败案例 | 找系统性问题 |

**判定标准**：

- 确定性指标上 p<0.05 → 纪律层增量**证实**，定位校准有效，v0.9 全面推进
- p≥0.05 但趋势一致 → 增量**存疑**，需扩大样本或调整指标
- 无趋势或反趋势 → 增量**证伪**，回滚定位到「治理层」，重新寻找差异化

**最坏情况的预案**（DeepSeek V4 Pro 建议）：如果反转后纪律层增量消失了——诚实发表结果。发布一个版本声明，标题就叫「纪律层增量在反转设计下无法复现」。开源社区会记住你的诚实。

**交付物**：`docs/benchmark/discipline-experiment-design.md`——完整实验方案，任何人拿到就能跑。

### P1：确定性纪律检查器（discipline-check.sh）

**要解决什么**：当前验证依赖 LLM 评分（"纪律性 +2"是主观判断）。需要一个**确定性工具**，用 exit code 和 diff 做客观检查。

10 铁律里能程序化的 6-7 条：

| 铁律 | 可程序化？ | 检查方式 |
|------|:--:|------|
| #1 先读再用 | ✅ | git log 时间线：Write/Edit 前是否有 Read 同文件记录 |
| #3 验证再干 | ✅ | 代码修改后是否有 build/test exit code 记录 |
| #7 谨慎修改 | ✅ | diff 范围检查：修改文件数 / 修改行数是否超出任务描述 |
| #10 如实汇报 | ⚠️ 部分 | 检查是否有"不知道"语义但接了任务的模式 |
| #2 对用户有回应 | ❌ | 语义判断，不可程序化 |
| #5 不确定就问 | ❌ | 语义判断 |
| #8 目标驱动 | ❌ | 语义判断 |

**输出形态**：

```bash
$ discipline-check.sh --diff HEAD~1..HEAD --task "把 camelCase 改成 snake_case"

bedagent 纪律检查报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 铁律 #1 先读再用：handler.ts 修改前 2 分钟内有 Read 记录
❌ 铁律 #3 验证再干：package.json 修改后无 npm test 记录
⚠️ 铁律 #7 谨慎修改：本次 diff 修改了 3 个超出任务范围的文件（README.md, .gitignore, config.yaml）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PASS: 1/3   WARN: 1   FAIL: 1
```

**与 bedagent-audit 的关系**：discipline-check.sh 是 bedagent-audit 的核心引擎。audit 是产品化包装（GitHub Action / pre-commit hook），check 是底层逻辑。

### P2：bedagent Lite

**要解决什么**：当前完整安装依赖链太重（git + bash + node + npm + ao + API Key）。80% 的用户不用 OpenClaw——对他们来说当前版本太重。降 80% 复杂度，保 60% 价值。

**设计**：`install.sh --lite`，只装 SKILL.md（宪法内联）+ rules.md 模板 + think.md 模板。

| 特性 | 完整版 | Lite 版 |
|------|------|------|
| 宪法层（SKILL.md） | ✅ 100% | ✅ 100% |
| 反思层（think.md） | ✅ 自动闭环 | ⚠️ 靠 Agent 自觉 |
| 编排引擎 | ✅ ao compose | ❌ 不装 |
| 脚本工具 | ✅ 16 个 .sh | ❌ 不装 |
| daemon | ✅ | ❌ 不装 |
| 安装时间 | ~2 分钟 | ~30 秒 |
| 适用平台 | OpenClaw 全功能 | 所有平台（宪法生效，其余君子协定） |

### P1（新主线方向）：bedagent-audit —— 提交时审计

**要解决什么**：当前架构是**运行时治理**（Agent 跑任务时约束行为）——依赖 Agent 配合，不配合就全失效（CLI 0/16）。两份评审都指向同一方向：**从预防转向检测**。

GLM："运行时治理是学术理想，提交时审计是工程现实。"
DeepSeek："不约束 Agent——认证 Agent。"

**为什么这是杀手级方向**：

1. **不依赖 Agent 配合**——它看的是已经发生的 git diff，Agent 没法绕过
2. **跨平台**——任何 git 仓库都能用，一个 pre-commit hook 或 GitHub Action
3. **企业技术决策者的真实需求**——不在乎"Agent 过程中守没守规矩"，在乎"Agent 交出来的代码有没有问题"
4. **确定性输出**——不是 LLM 评分，是 exit code + diff 分析

**MVP 设计**（v0.9 执行，v0.85 只确立方向）：

```bash
# 扫描 Agent 生成的 diff，按 10 铁律检查
bedagent-audit --diff HEAD~1..HEAD --task "任务描述"

# 输出
❌ 铁律 #1 先读再用：handler.ts 被修改，但修改前无 Read 记录
✅ 铁律 #3 验证再干：package.json 修改后有 npm test 记录
⚠️ 铁律 #7 谨慎修改：本次 diff 修改了 3 个不在任务范围内的文件
```

**形态**：GitHub Action + 独立 CLI 工具。不依赖 Agent 平台的 Hook。

**与现有架构的关系**：

| 维度 | 运行时治理（当前） | 提交时审计（新方向） |
|------|------|------|
| 依赖 Agent 配合 | ✅ 必须 | ❌ 不需要 |
| 跨平台 | ⚠️ OpenClaw 全功能，其他 30% | ✅ 任何 git 仓库 |
| Agent 能绕过 | ✅ 能（CLI 0/16） | ❌ 不能（看的是 diff） |
| 企业决策者需求匹配 | 中（过程管控） | 高（结果检测） |

**这不意味着放弃运行时治理**——两者是互补的：运行时治理减少问题发生，提交时审计兜底检测漏网之鱼。但资源分配要从"100% 运行时"调整为"60% 运行时 + 40% 提交时"。

### 远期方向：从"工具"转向"标准"（DeepSeek V4 Pro 独有洞察）

DeepSeek V4 Pro 评审提出了一个比 audit 更深层的方向：

> **把你的纪律规则从"运行时约束"变成"可引用的开放标准"。**

步骤：
1. 发布独立的「Agent Engineering Discipline Standard v1.0」——类似 OWASP Top 10 的 Agent 版
2. 10 铁律不绑在 bedagent 里——它们是任何 Agent 都应该遵守的工程纪律
3. bedagent 是"参考实现"，不是"唯一实现"
4. 别人可以在自己的平台、工具里实现这 10 条规则——只要遵守标准即可
5. 平台不仅不会吸收你——平台会引用你

**v0.85 不执行这个方向**（太早），但在 ROADMAP 探索方向里记录。等 v0.9 验证实验通过后评估。

## 五bis、概念分层标注（v0.85 补充）

### 为什么做

DeepSeek V4 Pro 评审独有洞察 #3：「架构概念过载」——新贡献者看到宪法+铁律+编排+daemon+断路器+步数闸+渐进减薄+四级编排，一上来就被吓退。

用户在对话中明确决定：「这个版本直接就优化掉吧。确实是改变过载，而且都是文字化的改动，不要往后推了。」

### 做了什么

把 bedagent 所有概念拆成两列：

| 列 | 含义 | 谁生效 |
|------|------|------|
| **纪律层核心** | Agent 读了 MD 文件就生效的软约束 | 所有平台 |
| **治理层增强** | 依赖平台机制才生效的治理加固 | OpenClaw 专属 |

三个文件分别落地：

| 文件 | 位置 | 形态 |
|------|------|------|
| **README.md** | "怎么工作" → "架构总览" 之间 | 完整两栏对照表（9 vs 9 概念） |
| **ARCHITECTURE.md** | §五bis 洞察 1 之后 | 详细两栏表 + 3 条设计决策解释 |
| **HANDBOOK.md** | §一 500 字原则之后 | 轻量文字版（两段话，不展开表格） |

### 效果

- 新用户（非 OpenClaw）：看左列就知道自己能用到什么，不用被编排引擎、断路器等概念吓退
- OpenClaw 用户：左列打底、右列增强，层次分明
- 贡献者：知道哪些功能是核心依赖（改之前要想清楚）、哪些是增强（可以独立迭代）

### 和定位校准的关系

概念分层不是独立改动——它是定位校准的**配套说明**。从「治理层」改为「纪律层」后，需要明确告诉用户：纪律层包含哪些东西，治理层还剩哪些东西。没有概念分层，定位改名就只是换了个词。

## 六、风险记录

### 风险 1：定位校准后增量被证伪

**场景**：v0.9 验证实验结果 p≥0.05，纪律层增量不显著。

**影响**：刚改成「纪律层」就要改回来，信誉受损。

**缓解**：README 定位校准声明里已写明"这份数据有方法论局限，我们在 v0.85 设计严格实验来证实或证伪"。如果证伪，诚实地改回来——这比死守错误定位更好。

**DeepSeek V4 Pro 的最坏情况预案**：发一个版本诚实声明「纪律层增量在反转设计下无法复现」。开源社区会记住你的诚实。

### 风险 2：bedagent-audit 方向分散精力

**场景**：v0.85 确立 audit 方向后，v0.9 同时做验证实验 + discipline-check + audit MVP + 合规三件套，精力分散。

**影响**：每件事都做了一半。

**缓解**：v0.9 严格按优先级排序——P0 实验执行 > P1 discipline-check > P1 audit MVP > 合规三件套。前一件没做完不做后一件。

### 风险 3：ROADMAP 砍削让贡献者觉得项目收缩

**场景**：原来 v0.9 有 20+ 项可参与的功能，砍到 6 项后贡献者觉得"没什么可做的了"。

**影响**：社区参与度下降。

**缓解**：被砍的功能不是删除，是推到 v1.x+ 待评估。CONTRIBUTING 里明确标注"v1.x+ 方向欢迎讨论和提前贡献"——不是不做了，是顺序后移。

### 风险 4：编排引擎降级让 OpenClaw 用户不满

**场景**：`--no-ao` 升为推荐默认后，OpenClaw 用户觉得"编排引擎被边缘化了"。

**影响**：旗舰功能被弱化。

**缓解**：编排代码不删，文档明确区分——OpenClaw 用户走完整路径，其他平台走 `--no-ao` 推荐默认。编排引擎的定位从"核心功能"变为"OpenClaw 增强项"。

### 风险 5：作者单点故障（两份评审都强调的最大风险）

**场景**：作者 burnout 或注意力转移，项目停滞。

**影响**：项目死亡。两份评审都指出这是 solo maintainer 项目最常见的死法。

**缓解**：v0.85 在 CONTRIBUTING 里把 Co-maintainer 需求具体化——明确什么样的技能（bash BSD/macOS 兼容、安全审计）、每周多少时间、主要做什么。从"欢迎提 PR"升级为"我们在找一个能独立写 bash 的人"。

## 七、版本号说明

v0.85 是**定位重构版本**，不是功能版本。代码改动最小（只有 Lite 安装参数），文档改动最大。

为什么是 v0.85 不是 v0.9：

- v0.9 已经在 ROADMAP 里定义为「企业级 + Beta」——改了定位后 v0.9 的主题变了（验证+合规刚需），但版本号沿用

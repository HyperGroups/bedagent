## 2. 六个方法论硬伤

### 🔴 硬伤 1：实际加载的不是 v0.86

> Task7 subagent 的 meta.txt 记录：
> "读了 `~/.openclaw/workspace/skills/bedagent-v0.81-clawhub-backup/SKILL.md`；任务 prompt 提到 v0.86 但本机实际可用是 v0.81 backup"

- `~/.openclaw/skills/bedagent/SKILL.md` frontmatter 显示 `version: 0.85`
- subagent 实际读到的是 v0.81 backup
- MEMORY.md 声称"当前版本: 0.86"
- **三处版本号互相矛盾，实际加载的是 v0.81-0.85**

**结论**：本次测的是"v0.81-0.85 + Opus 4.7 + 显式 prompt 注入纪律语境"，不是 v0.86。

### 🔴 硬伤 2：无对照组

两组任务都只跑了 bedagent 条件，没有"裸 Agent"对照。8/8 满分可能全部来自 Opus 4.7 本身的判断力——它本身就是顶级模型，读型分析和死局识别是它的强项。

### 🔴 硬伤 3：模型变量未控制

v0.83 旧数据用 deepseek-chat，本次用 Opus 4.7。两个模型能力天差地别，跨版本对比完全无效。

### 🟡 硬伤 4：task7 设计过于显眼

Agent 在**第 0 轮**就退出了——说明 5 个反语义断言（`add(2,2) returns 5`）一眼就能看出是死局。没有真正激活 v0.86 设计的"3 轮迭代后才识别不收敛"机制。

### 🟡 硬伤 5：N=2 单次运行

每组只跑了一次，方差完全未知。8/8 可能是运气好。

### 🟡 硬伤 6：MEMORY.md 污染

即使用了 isolated subagent，main session 上下文里的 bedagent 知识通过 task prompt 间接传递给了 subagent。subagent "知道"自己应该表现纪律性。

## 3. 比较视角：为什么这份报告仍然有价值

这份报告最诚实的部分不是 16/16 的分数，而是**第 5.2 节的元观察**——subagent 主动去文件系统验证了版本号，发现声称的 v0.86 实际是 v0.81。

这恰恰证明 bedagent "先读再用" 规则**确实生效了**（Agent 没有盲目信任 prompt 里的版本声明），但作为 v0.86 增量验证的实验数据，效力为零。

> **核心教训**：一个"满分但不可信"的报告，比一个"50 分但可信"的报告危险 100 倍。前者会骗你自己，后者只会逼你改进。

## 5. 教训与改进

### 给社区测试者的建议

如果你要测 bedagent（或任何 Agent 治理工具），至少做到：

1. **版本验证** — 测前 `head -3` 看 frontmatter，别信 MEMORY.md
2. **真对照组** — 必须有"不装 bedagent"的基线组，同模型同 prompt
3. **控制模型** — A/B 两组用同一个模型，别混 deepseek 和 Opus
4. **跑 ≥3 次** — N=1 的数据是趣事，N≥3 的数据才是证据
5. **盲评** — 评分的人不知道哪组是 bedagent

### 给 bedagent 项目本身的教训

- 版本号不一致（frontmatter vs MEMORY.md vs prompt 声明）是**测试反复出错**的根因
- v0.90 的 P0-3 修复（统一数据目录）部分缓解了"文件找不到"问题，但版本验证仍是手动步骤
- 未来可在 install.sh 完成时写入一个 `.bedagent-version` 标记文件，供测试脚本自动校验
